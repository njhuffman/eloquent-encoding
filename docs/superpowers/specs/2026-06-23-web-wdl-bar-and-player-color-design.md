# Web WDL bar + player color — design

**Status:** approved design; precedes the implementation plan.

## Goal

Add two bundled features to the GitHub Pages bot:

1. A **WDL bar** — a fixed-size vertical bar left of the board, split into three segments sized by
   P(white win) / P(draw) / P(black win) for the current position, refreshed on every move/undo/new
   game.
2. **Play as either color** — a White/Black picker; choosing a color starts a fresh game, the board
   orients to the player's side, and the WDL bar is arranged so **the player's win-probability is the
   bottom segment** (matching the flipped board).

The WDL bar needs a value head, which only exists in the `wdl_16M` checkpoint (not in the deployed
`base_64M` policy graphs). Per the model-source decision, we **switch the deployed bot to `wdl_16M`**
(one self-consistent model: policy + value), exporting a value-head graph alongside the policy graphs.

## Background / current state

- Deployed graphs (`web/public/{encode,from_head,to_head}_int8.onnx`) come from `base_64M` and have
  **no value head**. The `Engine` (`web/src/inference/engine.ts`) loads those three, encodes a board
  to `squares`, and runs the two policy heads. The encoder actually returns `(cls, squares)` but
  `EncodeExport` (`style_policy/onnx_export.py`) discards `cls`.
- `wdl_16M` (`style_policy_checkpoints/wdl_16M/wdl_16M_stage_1.pt`): `d_model=256`, `head_hidden=512`,
  `elo_dim=32`, `n_elo_buckets=40`. Its `value_head` (`style_policy/value_head.py` `WDLHead`) maps
  `cls (B,256) + elo_idx → 3 logits` in the order **(loss=0, draw=1, win=2)** from the **side-to-move's**
  perspective. `BasePolicy.forward_value` = `value_head(encode(packed)[0], elo_idx)`.
- `BoardPanel.tsx` currently **hardcodes the human as White**: `turn() === "w"` gates the "Your move"
  panel, the "Bot's last move" panel reconstructs the last **Black** move, the board uses the default
  (white-at-bottom) orientation, and the bot always plays Black.
- `elo` and `temperature` state live in `App.tsx` and flow into `BoardPanel` via props; `Controls.tsx`
  renders the sliders.
- Parity is gated by `web/src/inference/engine.node.test.ts` against
  `web/src/inference/__fixtures__/cases.json`, generated by `scripts/gen_web_fixtures.py`. ONNX export
  is `scripts/export_onnx.py` (fp32 export → int8 `quantize_dynamic`).

## Components

### 1. ONNX export: add `cls` output + value-head graph (build side)

**`style_policy/onnx_export.py`:**
- `EncodeExport.forward` returns **both** `squares` and `cls` (encoder already computes both):
  `cls, squares = self.encoder(board_tensor); return squares, cls`. Output names `["squares", "cls"]`.
  Backward-compatible for the policy path (it reads `out["squares"]`).
- Add `ValueHeadExport(nn.Module)` wrapping `policy.value_head`:
  `forward(self, cls, elo_idx) -> value_logits` = `self.value_head(cls, elo_idx=elo_idx)`.
- `build_export_modules` returns the value-head module too.

**`scripts/export_onnx.py`:**
- Export four graphs: `encode.onnx` (`board_tensor → squares, cls`), `from_head.onnx`, `to_head.onnx`
  (unchanged signatures), and `value_head.onnx` (`cls, elo_idx → value_logits`). Trace `value_head`
  with a `cls` example obtained from `enc(bt)` (now returns `(squares, cls)`).
- `quantize_and_check` quantizes all four (`encode`, `from_head`, `to_head`, `value_head`).
- `model_meta.json` `files` list includes `value_head_int8.onnx`.
- Run it with `--checkpoint style_policy_checkpoints/wdl_16M/wdl_16M_stage_1.pt`; commit the four
  regenerated `web/public/*_int8.onnx` + `model_meta.json`.

### 2. Parity fixtures: regenerate from `wdl_16M` + value case (build side)

**`scripts/gen_web_fixtures.py`:** for each fixture FEN also compute the value:
`cls, sq = policy.encoder(bt); value_logits = policy.value_head(cls, elo_idx=...)`. Add
`"value_logits": value_logits[0].tolist()` (length-3, loss/draw/win) to each case. Regenerate
`cases.json` from `wdl_16M` (`--checkpoint .../wdl_16M_stage_1.pt`); the policy logits/`top_move_uci`
change with the model — expected, the parity test re-pins to the new model.

### 3. `Engine.value` (web)

**`web/src/inference/engine.ts`:**
- `Engine.load` also creates the `value_head` session; constructor stores it. `urls` gains
  `valueHead: string`.
- `squares()` becomes an internal encode that returns both tensors (rename/extend to return
  `{ squares, cls }` from `out["squares"]` / `out["cls"]`); `policy`/`chooseMove`/`distributions`
  keep using `squares` exactly as before.
- `async value(board: Chess, elo: number): Promise<{ loss: number; draw: number; win: number }>`:
  encode → `cls`, run `value_head` with `{ cls, elo_idx }`, softmax the 3 logits, return
  `{ loss, draw, win }` (**side-to-move perspective**, order loss/draw/win). Uses the same
  module-global `serializedRun` queue as every other run.
- **`web/src/useEngine.ts`** (the caller) passes the new `valueHead` URL to `Engine.load` (alongside
  the existing `encode`/`fromHead`/`toHead`), pointing at `value_head_int8.onnx`.

### 4. Player color state + control (web)

- **`App.tsx`:** add `const [playerColor, setPlayerColor] = useState<'w' | 'b'>('w')`; pass both into
  `Controls` and `BoardPanel`.
- **`Controls.tsx`:** add a White/Black selector (two buttons or a segmented control) bound to
  `playerColor`/`setPlayerColor`. Changing it is what triggers a new game (handled in `BoardPanel`
  via a prop-change effect, below) — the control itself just sets state.

### 5. Generalize `BoardPanel` for player color (web)

Replace the hardcoded-White assumptions with `playerColor`:
- `boardOrientation = playerColor === 'w' ? 'white' : 'black'` on `<Chessboard>`.
- "Your move" panel computes when `!isGameOver() && g.turn() === playerColor`.
- "Bot's last move" reconstructs the last move of the **opposite** color
  (`botColor = playerColor === 'w' ? 'b' : 'w'`; scan history for the last `color === botColor`).
- `onDrop` only allows dragging on the player's turn (`g.turn() === playerColor`); after the human's
  move it calls `botMove()` (unchanged).
- **New game on color change / New-game button:** reset `gameRef` to `new Chess()`; if
  `playerColor === 'b'`, immediately `void botMove()` so the bot (White) opens. Implement via an
  effect keyed on `playerColor` that resets + (for Black) kicks the bot, and have the existing
  `newGame` button do the same reset-then-maybe-bot sequence. Guard against double-fire (don't start
  a bot move if one is already in flight / `thinking`).

### 6. `WDLBar` component (web)

**`web/src/components/WDLBar.tsx`** — pure presentational + a tiny arrangement helper:
- Props: `wdl: { loss: number; draw: number; win: number } | null` (side-to-move perspective),
  `sideToMove: 'w' | 'b'`, `playerColor: 'w' | 'b'`.
- Convert STM-perspective → white/black:
  `pWhite = sideToMove === 'w' ? wdl.win : wdl.loss`, `pBlack = sideToMove === 'w' ? wdl.loss : wdl.win`,
  `pDraw = wdl.draw`.
- Arrange segments top→bottom so the **player's color is at the bottom**:
  - bottom = player's color win prob, top = opponent's win prob, draw in the middle.
  - i.e. if `playerColor === 'w'`: top=black, mid=draw, bottom=white; if `'b'`: top=white, mid=draw,
    bottom=black.
- Render a vertical bar (~28px wide, height = board height 480px) of three stacked divs with
  `flex` sizes ∝ the three probs; colors: white-win = light (`#f0f0f0`), black-win = dark (`#333`),
  draw = neutral gray (`#9e9e9e`); show each segment's `%` (small, centered) when its size is big
  enough to fit. `null` wdl → a flat placeholder (e.g. all-gray or empty) — bar still occupies space.
- Layout: rendered as the **left** child of the board row in `BoardPanel` (board row becomes
  `[WDLBar][board+controls][thinking panels]`).
- The WDL value is computed in `BoardPanel` in the **same effect** that recomputes the thinking
  panels (keyed on `engine, fen, elo`): `engine.value(new Chess(g.fen()), elo)` → store in state →
  pass to `WDLBar` along with `g.turn()` and `playerColor`. Conditioning on the **elo slider** is a
  deliberate simplification (exact when it's the bot's turn; an approximation on the human's turn,
  whose true rating we don't model).

## Data flow

```
position changes (move / undo / new game / color change):
  effect (engine, fen, elo):
    yourMoves   = turn===playerColor ? topMoves(...) : []
    botAnalysis = reconstruct last opposite-color move + its topMoves
    wdl         = engine.value(currentBoard, elo)        # {loss,draw,win}, STM perspective
  render:
    <WDLBar wdl sideToMove=turn playerColor />           # player color at bottom
    <Chessboard boardOrientation=playerColor ... />
```

## Testing

- **Value parity** (`engine.node.test.ts`, extend or sibling test): for each `cases.json` entry,
  `engine.value(new Chess(c.fen), c.elo)` softmax ≈ softmax(`c.value_logits`) within tolerance
  (same byte-tolerance approach as the existing top-move parity). Existing greedy-top-move parity
  must still pass against the regenerated (wdl_16M) fixtures.
- **`WDLBar` unit test** (no ORT): given a WDL triple + `sideToMove` + `playerColor`, the helper
  produces correct segment sizes and the correct top/mid/bottom arrangement —
  `playerColor==='w'` → white at bottom; `playerColor==='b'` → black at bottom; STM flip is applied.
- **`BoardPanel` test** (stubbed engine, as in existing tests): picking Black resets to a fresh game
  and triggers exactly one bot (White) opening move; the human can only move on `playerColor` turns.

## Out of scope

- Conditioning the value on the *actual* side-to-move's rating (we use the elo slider as a proxy).
- Numeric/centipawn eval or a single-value eval bar (this is a 3-way WDL bar).
- Animating bar transitions; showing WDL history/sparkline.
- Mid-game side-swapping (color change always starts a new game).
- A book/threshold or value on/off UI control.

## Risks

- **Parity drift from the model switch:** regenerating `cases.json` from `wdl_16M` re-pins both the
  policy top-move parity and the new value parity; CI/test catches a bad export.
- **Bot strength change:** switching to `wdl_16M` weakens move-match (~42% vs base_64M ~50%);
  accepted (openings are still book-driven; it's a demo bot).
- **Double bot-move on color change:** the `playerColor` effect + `newGame` could both fire a bot
  move — guarded by the `thinking` flag and a single reset path.
- **`cls` output change breaking the policy path:** mitigated because the policy reads `out["squares"]`
  by name; adding a second named output is non-breaking, and the parity test verifies the policy path.
